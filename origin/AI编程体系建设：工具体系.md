# AI编程体系建设：工具体系

本文档是 [AI编程方法论：骑士、缰绳和千里马](./AI编程方法论：骑士、缰绳和千里马.md) 的进一步细化，将理论框架落地为可操作的工具体系。

---

## 方法论核心（精炼提取）

### 三个角色

AI编程可以抽象为三个角色：

- **骑士（开发者）**：指挥官，决策者，最终责任人。拥有对项目的心智模型，是唯一有权修改缰绳的人。
- **缰绳（文档）**：骑士、AI、代码三者之间的介质，兼具约束（单向，规定边界）与信息交换（双向，达成共识）两种特性。
- **缰绳的不完备性**：缰绳是引导工具，帮助AI快速定位，避免大量阅读代码却找不到位置。有文档就用，没有文档就退化到纯代码探索模式。文档缺失时，AI应该继续工作，而不是纠结为什么没有文档。
- **千里马（AI）**：在约束范围内执行的行动者。拥有强大执行力但缺乏方向感，无法自我评估，上下文窗口有限。

### 核心机制

- **对齐**：通过文档让骑士和 AI 对项目达成共同理解，消除方向偏差。对齐是双向的，不是单方面指令。
- **行业标准**：约束文档不是凭空制定的，而是在实践中逐步演化。先有超越标准的实践，才会有标准的提升。
- **共演化**：骑士、缰绳、千里马三者相互磨合、共同进化。成熟领域可以半自治，新领域需要紧密控制。
- **回合制**：小回合（一次输入一次输出）是最小操作单元，若干小回合组成一个大回合（完整功能交付），项目在大回合中前进。
- **执行日志**：每个小回合结束后，AI 总结遇到的问题和困难，为骑士提供决策依据。
- **渐进式披露**：每个阶段读取最需要的文档，辅助类信息尽可能传递，避免上下文窗口过载。

### 核心原则

1. 方向优先于速度
2. 对齐优先于能力
3. 缰绳是活的，在共演化中持续进化
4. 质量可迁移——缰绳是模型无关的资产
5. 骑士是唯一有权修改缰绳的人
6. 生成与评估分离
7. 提示词会过时，缰绳不会

## 工具体系

基于上述方法论，工具体系分为四层：

```
文档层    —— 建立缰绳（lb-doc-owner，协调者+初始化子代理+并行维护子代理）
执行层    —— 代码编写（ooda-coder，内置执行日志）
校验层    —— 质量保障（独立审查 + 架构守护 + 问题诊断）
进化层    —— 共演化循环（consolidation 沉淀分析，为开发者提供决策依据）
```

## 文档层

### lb-bootstrap（初始化子代理）

**职责**：项目初始化。由 lb-doc-owner 协调者调度，当项目文档（INDEX）不存在时触发，快速搭建文档框架。

**触发时机**：lb-doc-owner 检测到 INDEX 不存在时自动调度。

**两种场景**：

#### 场景一：新项目（从零开始）

**角色**：引导者。通过提问了解项目层级的问题，搭建最原始的项目层文档骨架。

**工作内容**：
- 提问了解项目：做什么、解决什么问题
- 确认技术选型：语言、框架、数据库等
- 确认架构风格：单体、微服务、前后端分离等
- 确认编码风格偏好（是否有参考规范）

**输出**：
- 项目介绍文档（骨架，不确定的留空）
- 编码规约（根据技术选型提供模板，骑士选择/修改）
- 项目级 INDEX（空，建议定义执行日志存放位置；如果骑士不愿意建立文件夹，后续 ooda-coder 可输出到控制台）
  - INDEX中应包含"AI执行记录"段落，配置路径为`.ai-work-logs/`

**注意**：不涉及业务模块划分，那是后续骑士在推进功能时逐步建立的。

#### 场景二：接手已有项目

**角色**：分析师。分析项目现状，输出分析报告，等骑士确认后再决定创建什么文档。

**工作内容**：
- 扫描项目结构，识别技术栈、目录组织
- 分析代码风格（命名、格式、注释习惯）
- 分析模块划分（哪些是基础设施、哪些是业务、哪些边界模糊）
- 识别公共工具函数

**输出**：
- 分析报告（不直接创建文档）

**后续流程**：
- 骑士审阅分析报告
- 骑士确认需要创建哪些文档
- 按确认内容创建，尽量少创建，避免文档爆炸
- 固定必要的：项目介绍、编码规约、项目级 INDEX
- 其他：视项目实际情况而定

**与 lb-doc-owner 的关系**：bootstrap 是 lb-doc-owner 的初始化子代理，负责"从无到有"；lb-doc-owner 的维护子代理负责"从有到优"。两者由 lb-doc-owner 协调者统一调度。

---

### lb-doc-owner

**职责**：统一维护项目三层架构（项目层、业务层、基础设施层）的全部文档，包括初始化（当文档不存在时）和维护（当文档存在时）。

**架构概览**：

lb-doc-owner 通过协调者 + 初始化子代理 + 2 个并行维护子代理实现统一管理。

| 角色 | 职责 |
|------|------|
| 协调者 | 检测项目状态、读取 INDEX、解析边界、调度子代理、维护项目级文档、生成执行日志 |
| 初始化子代理（lb-bootstrap） | INDEX 不存在时触发，搭建文档骨架 |
| 业务层子代理 | 扫描业务模块，维护已存在的业务层文档 |
| 基础设施层子代理 | 扫描基础设施模块，维护已存在的基础设施层文档 |

**核心原则**：
- 代码是唯一事实来源，文档与代码不一致时以代码为准
- 已存在的文档必须维护，不存在的文档不得擅自创建
- 项目级文档的任何变更必须经开发者确认
- 全局编码规约是项目基准，修改需开发者明确授权

**行为准则（协调者注入）**：
- 路径规范：所有路径基于项目根目录
- 决策矩阵：代码清晰时不询问，代码有歧义时才询问开发者
- 禁止事项：不创建缺失文档、不越界操作、声称一致必须附证据

**INDEX 导航规范**：
- 项目级 INDEX → 基础设施层全局 INDEX → 模块 INDEX（接力模式）
- 项目级 INDEX → 业务模块目录（至少可定位到 README 或直接探索代码）
- INDEX 保持精简，每个模块条目包含：模块名、目录路径、一句话说明
- 有 README 的模块 INDEX 只需定位，无 README 的模块 INDEX 提供一句话说明

**执行日志**：

日志目录结构（文件夹形式）：
```
{UTC紧凑时间戳}-文档维护/
├── business-report.md     # 业务层子代理报告
├── infra-report.md        # 基础设施层子代理报告
└── execution-log.md       # 执行日志汇总
```

日志路径读取项目级 INDEX 中"AI执行记录"段落的配置。

## 执行层

### ooda-coder

**职责**：基于 OODA-E 循环进行代码编写。

**核心价值观**：规范大于自由，基建优先于手写，质量兜底一切。

**架构概览**：

ooda-coder 通过 4 个 subagent 实现上下文隔离，确保弱模型也能保持最优能力。

| 角色 | OODA-E阶段 | 职责 |
|------|-----------|------|
| ooda-scout | Observe+Orient | 读取文档，输出上下文快照 |
| ooda-forger | Decide+Draft+Verify | 生成代码+静态预检 |
| ooda-act | Act | 写入文件+编译+基础功能验证 |
| ooda-eval | Evaluate | 回归测试+覆盖率分析+补充测试 |
| 协调者 | 执行日志 | 调度子代理+注入行为准则+汇总过程+识别卡点+询问用户 |

**行为准则（协调者注入）**：
- 路径规范：所有路径基于项目根目录的相对路径，禁止片段路径
- 不猜测原则：不确定时列为待确认事项，询问开发者
- 提问规范：结构化选项，用（推荐）标注推荐项

**文件链路**：

```
context-brief.md → code-plan.md → execution-report.md → verify-result.md
     (scout)         (forger)          (act)               (eval)
```

每个子代理可访问链路中所有前置输出，默认只读取直接输入，需要时沿链路回溯。

**临时文件机制**：

协调者负责临时文件的完整生命周期。每次执行创建唯一sessionId（UTC紧凑时间戳），临时文件存放在`.ooda-work/{sessionId}/`。

协调者创建目录、分配写入路径、管理清理。子代理只负责写入，不管理目录。

| 环节 | 写入 | 读取 |
|------|------|------|
| 协调者 | 创建/移入日志目录 | - |
| ooda-scout | context-brief.md | - |
| ooda-forger | code-plan.md | context-brief.md |
| ooda-act | execution-report.md | code-plan.md（默认），context-brief.md（自修复） |
| ooda-eval | verify-result.md | execution-report.md（默认），code-plan.md + context-brief.md（按需回溯） |

降级策略：子代理写入失败时，在返回中输出完整内容。协调者根据返回值判断传递方式。

汇报原则：写文件是存档，汇报是通知。子代理写入文件后，必须在返回中汇报关键信息（待确认事项、BLOCKED、测试失败等），协调者依赖汇报做决策，不会主动读取文件内容。

正常完成时，临时目录整体移入日志目录，成为执行日志的一部分。

**OODA-E 流程**：

1. **ooda-scout（侦察）**：业务上下文与基建环境侦察
   - 必读项目索引文件（INDEX或等效文件），获取项目结构和模块位置
   - 如果项目索引文件不存在，按降级链读取：README → 根目录md文件 → 扫描代码目录结构
   - 根据任务复杂度和意图清晰度判断是否需要额外读取README/架构文档
   - 读取业务文档，获取业务上下文
   - 编码规约只记录路径，不读取内容
   - 一切以代码为准，文档与代码有差异时只提示不修改
   - 文档缺失时继续探索代码，只提示不阻塞

2. **ooda-forger（规划）**：规范与依赖对齐，制定执行计划
   - 从CONTEXT-BRIEF获取项目结构、规约路径、业务背景、架构约束
   - 必读编码规约内容
   - 从代码中提取依赖的API签名
   - 拟稿阶段，不能完全相信文档描述，依赖的功能必须查询代码确认
   - 如果对业务无法理解，必须停下来询问开发者

3. **ooda-act（执行）**：代码编写与验证
   - 从CODE-PLAN获取文件路径、完整代码、验证命令
   - 默认模式（机械执行）：只读取CODE-PLAN，不读取其他文档
   - 自修复模式（编译失败时激活）：可读取CONTEXT-BRIEF获取规约路径和项目结构，探索范围仅限于修复当前编译错误所需
   - 写入代码并执行编译检查
   - 如果编译失败，自行修正并重新验证，最多3次
   - 如果3次修正后仍失败，停止并报告
   - 输出execution-report.md到临时目录

4. **ooda-eval（评估）**：回归测试与覆盖率分析
   - 默认读取execution-report.md和测试代码
   - 需要时沿链路回溯读取CODE-PLAN和CONTEXT-BRIEF
   - 使用git diff获取实际变更的文件列表，结合用户意图进行交叉验证
   - 测试失败或交叉验证发现差异时，先分析原因：
     - 功能bug：自行修复
     - 测试样例过时：自行更新测试样例
     - 业务不清楚：询问开发者
     - 改动点明显影响其他功能：询问开发者
   - 小bug、逻辑问题自行解决，问题较大或不清楚时询问开发者
   - 输出verify-result.md到临时目录

**回溯机制**：

回溯是开发者的明确指令。当流程出问题停止时，协调者可以建议开发者使用回溯指令，但最终决定权在开发者。

- 回溯指令格式：`回溯到 [节点名称]`
- 可选节点：scout、forger、act、eval
- 回溯时保留之前阶段的输出（如CONTEXT-BRIEF）

**执行日志**：

全部子代理执行完毕后，协调者负责汇总各子代理的交互过程，生成执行日志。

日志目录结构（文件夹形式）：
```
{UTC紧凑时间戳}-代码编写/
├── context-brief.md       # scout 输出
├── code-plan.md           # forger 输出
├── execution-report.md    # act 输出
├── verify-result.md       # eval 输出
└── execution-log.md       # 执行日志汇总
```

读取项目级INDEX，查找"AI执行记录"段落中的"路径"配置：
- 有配置：将临时目录整体移入日志目录，不询问
- 无配置：询问开发者是否保存
  - 是：让开发者输入路径，移入，提示在INDEX上配置以后都会生效
  - 否：不记录

日志中用户输入必须完整记录，不得缩减。

**硬性规定**：
- 如果 AI 认为需要改动骑士未明确说明的相关地方，必须经过骑士同意
- 禁止手写基础设施已有的能力，必须调用 INDEX 中提供的 API
- 所有子代理无权修改文档，发现不一致时汇报给协调者
- 所有路径必须使用基于项目根目录的相对路径，禁止片段路径

**与文档框架的关系**：
- ooda-coder 在文档框架的约束下执行，无权修改文档
- AI 发现文档有误时，不能自行修改，而是总结不同，让骑士决策
- 行为准则由协调者注入，确保所有子代理遵循统一规范

**文档读取策略**：

渐进式披露，每个阶段读取最需要的文档，辅助类信息通过文件链路传递：

**设计原则**：
- INDEX（方向标）贯穿始终，每个阶段都需要知道项目结构
- 编码规约在侦查阶段只记录路径，规划阶段读取内容，编码阶段默认不读取（编译失败时按需读取）
- 文件链路（context-brief → code-plan → execution-report）实现渐进式披露，每个子代理默认只读直接输入，需要时沿链路回溯

| 文档类型 | ooda-scout | ooda-forger | ooda-act | ooda-eval |
|----------|------------|-------------|----------|-----------|
| INDEX（方向标） | 读取 | 从快照获取 | 从快照获取 | 从快照获取 |
| 编码规约 | 只记录路径 | 读取内容 | 默认不读取，编译失败时按需读取 | 读取内容 |
| README/架构文档 | 按需读取 | 不需要 | 不需要 | 不需要 |
| 目标文件 | 不需要 | 不需要 | 读取 | 读取 |
| 测试代码 | 不需要 | 不需要 | 不需要 | 读取 |

文件树优化：
- 有全局INDEX时：基础设施层指向全局INDEX，业务层指引到涉及的模块目录
- 无全局INDEX时：指引规划agent到正确的目录位置

## 校验层

### reviewer

**职责**：独立审查 ooda-coder 的产出，提供客观的质量评估。

**核心原则**：
- 生成与评估必须分离
- 所有审查维度必须客观可验证
- 只输出报告，不做修改决定
**触发时机**：由开发者自行判断。

**工作流程**：
1. **独立 session 启动**：干净的上下文，不受 ooda-coder 的上下文影响
2. **读取项目文档**：README、INDEX、CONVENTIONS（全局 + 业务）
3. **读取代码变更**：获取 ooda-coder 的代码产出
4. **按维度审查**：逐项检查 14 个审查维度
5. **输出审查报告**：结构化报告，包含各维度结果和总体结论

**审查维度（14 个）**：

#### 基础合规维度
1. **编码规约符合度**：对照 CONVENTIONS，检查命名规范、代码风格、禁止用法等
2. **架构合规性**：检查依赖方向、模块边界是否符合架构约束
3. **基础设施使用**：检查是否正确调用 INDEX 中的 API，是否复用了已有能力
4. **业务逻辑一致性**：对照业务文档，检查代码中的业务逻辑是否一致

#### 质量维度
5. **代码质量**：可读性、可维护性、边界处理、异常处理等客观标准
6. **变更影响分析**：修改了哪些函数/类、这些函数/类在项目中的其他使用点、变更是否可能影响其他功能、是否需要更新相关测试
7. **可扩展性**：是否易于扩展，是否遵循开闭原则
8. **可测试性**：是否易于测试，是否依赖注入、可 mock
9. **性能考虑**：潜在性能问题，如 N+1 查询、不必要的循环
10. **安全性**：安全漏洞，如 SQL 注入、XSS
11. **错误处理**：异常处理是否恰当，是否有未捕获的异常
12. **技术债务**：是否引入技术债务，是否有临时解决方案
13. **设计模式使用**：是否使用了合适的设计模式，是否过度设计
14. **接口设计**：API 设计是否合理，是否易于理解和使用

**输出格式**：

```markdown
# reviewer 审查报告

## 审查摘要
- 审查范围：[文件列表]
- 审查时间：[时间]
- 总体结论：[通过/有条件通过/不通过]

## 各维度审查结果
[每个维度包含：状态、问题描述、严重程度]

## 变更影响分析
- 修改的函数/类：[列表]
- 其他使用点：[列表]
- 回归风险：[高/中/低]
- 建议的测试：[列表]

## 改进建议
- [建议列表]

## 总体结论
[通过/有条件通过/不通过] + 理由
```

**与骑士的关系**：reviewer 只输出报告，不做修改决定。修改决定由骑士做出。

**执行日志**：

审查完成后生成执行日志。发现问题时详细记录各维度审查结果；全部通过时简略记录总体结论。

日志目录结构（文件夹形式）：
```
{UTC紧凑时间戳}-审查/
├── review-report.md       # 审查报告
└── execution-log.md       # 执行日志汇总
```

日志路径读取项目级 INDEX 中"AI执行记录"段落的配置。

### architecture-guard

**职责**：架构约束的机械化检查，确保代码合入前符合架构规范。

**核心原则**：
- 有些约束不能只靠文档"建议"，必须被确定性地检查
- 检查规则来源于项目级文档和部门级文档
- 只输出报告，不做修改决定

**触发时机**：由开发者自行判断。

**工作流程**：
1. **读取架构约束**：从项目级文档和部门级文档中读取
2. **解析代码变更**：获取待合入的代码 diff
3. **执行检查规则**：按维度逐项检查
4. **输出检查报告**：通过/不通过 + 具体违规项

**检查维度（13 个）**：

#### 架构约束维度
1. **依赖方向**：检查 import 语句，确保没有反向依赖
2. **模块边界**：检查是否直接访问其他模块的内部类/方法
3. **分层架构**：检查调用链是否符合分层架构
4. **禁止用法**：检查是否使用了禁止的 API 或模式，安全检查（具体检查内容参考项目级编码规约）

#### 变更管理维度
5. **接口兼容性**：检查是否修改了现有的 API 接口（方法签名、返回类型等）
6. **数据库变更**：检查是否有数据库 schema 变更（如 DDL 语句）
7. **配置变更**：检查是否有配置文件变更
8. **依赖变更**：检查是否有新的依赖（如 pom.xml、package.json 变更）

#### 质量保障维度
9. **测试覆盖**：检查是否有对应的测试代码
10. **文档更新**：检查是否需要更新文档（如 API 文档、业务文档）

#### 文档规范维度
11. **重复文档检查**：检查是否有重复的文档内容
12. **文档位置检查**：检查文档是否放在正确的位置
13. **文档必要性检查**：检查是否创建了不必要的文档

**检查规则来源**：
- **依赖方向、模块边界、分层架构**：从 README 中的架构描述提取
- **禁止用法**：从 CONVENTIONS 中的禁止用法章节提取
- **安全检查**：从 CONVENTIONS 中的安全规约提取
- **接口兼容性**：从 INDEX 中的函数签名提取
- **数据库变更、配置变更、依赖变更**：从代码 diff 中识别
- **测试覆盖**：检查测试目录中是否有对应的测试文件
- **文档规范**：从 doc-owner 的文档规范提取

**输出格式**：

```markdown
# architecture-guard 检查报告

## 检查摘要
- 检查范围：[文件列表]
- 检查时间：[时间]
- 总体结论：[通过/不通过]

## 各维度检查结果
[每个维度包含：状态、违规项描述]

## 违规项详情
- [违规项]：[文件:行号] - [违规描述]

## 总体结论
[通过/不通过] + 理由
```

**与 reviewer 的区别**：
- reviewer 关注"代码写得好不好"——是质量审查
- architecture-guard 关注"代码有没有违反架构约束"——是合规检查

**与 lb-doc-owner 的关系**：architecture-guard 的检查规则来源于 lb-doc-owner 管辖的项目级文档。如果架构约束需要调整，由骑士授权 lb-doc-owner 修改。

**执行日志**：

检查完成后生成执行日志。发现违规时详细记录各维度检查结果和违规项；全部通过时简略记录总体结论。

日志目录结构（文件夹形式）：
```
{UTC紧凑时间戳}-架构检查/
├── guard-report.md        # 检查报告
└── execution-log.md       # 执行日志汇总
```

日志路径读取项目级 INDEX 中"AI执行记录"段落的配置。

### debugger

**职责**：系统化调试。当出现 bug、测试失败、构建错误等代码问题时，通过严谨的调查流程定位根因，输出诊断报告。

**核心原则**：
- 先定位根因，再提出修复方案
- 诊断与修复分离——只输出诊断结论，不做修改决定
- 调试是发现系统问题的最佳时机，执行日志为沉淀分析提供数据来源

**触发时机**：由开发者自行判断。出现代码缺陷、测试失败、构建错误、异常行为时触发。

**工作流程**：
1. **快速定位**：有文档看文档快速导航，没文档扫描代码目录
2. **根因调查**：仔细阅读错误信息、稳定复现、检查近期变更、追踪数据流
3. **模式分析**：找到可用示例、对比差异、理解依赖
4. **假设验证**：形成单一假设、最小化验证、科学方法
5. **诊断结论**：输出结构化诊断报告
6. **执行日志**：记录调试纪要，写入执行日志目录

**诊断报告**：
```markdown
# debugger 诊断报告

## 问题摘要
- 问题描述/复现步骤/复现率/影响范围

## 根因分析
- 故障点/根因/证据链/触发条件

## 模式分析
- 可用示例/关键差异/依赖关系

## 修复建议
- 建议方案/改动范围/风险评估/验证方式

## 待确认事项/总体结论
```

**执行日志**：

调试完成后生成执行日志，记录调试纪要（问题、根因、根因类型、调查耗时），为 consolidation 提供数据来源。日志路径读取项目级 INDEX 中"AI执行记录"段落的配置。

日志目录结构（文件夹形式）：
```
{UTC紧凑时间戳}-调试/
├── investigation.md       # 调查过程详细记录
├── diagnosis-report.md    # 诊断报告（或失败报告）
└── execution-log.md       # 执行日志汇总
```

调查过程记录（investigation.md）包含每一步操作、发现、判断，用于回溯排查路径。日志中用户输入必须完整记录，不得缩减。

**与 reviewer 的区别**：
- reviewer 审查的是"代码写得好不好"——在代码已经写完之后做质量审查
- debugger 诊断的是"代码为什么出了问题"——在问题发生后定位根因

**与 ooda-coder 的关系**：
- ooda-coder 负责写代码，debugger 负责诊断问题
- debugger 的诊断报告是 ooda-coder 修复问题的决策依据
- 两者都生成执行日志，共同为 consolidation 提供数据

## 进化层

### consolidation

**职责**：沉淀分析。分析执行日志积累，识别文档框架缺陷，为开发者提供决策依据。

**核心原则**：
- consolidation 是分析器，不是执行器
- 只分析、出报告，不改动任何代码和文档（除了分析索引）
- 从执行失误中提取文档框架的缺陷，而不是追究责任
- 输出是决策依据，由开发者决策，由 lb-doc-owner 执行
- 问题必须映射到框架的具体组成部分（INDEX、CONVENTIONS、Skill 定义、行为准则）

**触发时机**：由开发者自行判断。

**分析方法论**：

问题分类体系（7 类）：

| 分类 | 映射目标 | 示例 |
|------|---------|------|
| 文档质量 | INDEX / CONVENTIONS / README | 规约缺失、规约模糊、规约过时、导航缺陷 |
| Skill 设计 | Skill 定义 | 流程缺陷、输出格式缺陷、约束矛盾 |
| 能力边界 | 框架设计 | 上下文窗口限制、模型差异、推理能力边界 |
| 用户输入 | 无（改善输入习惯） | 需求模糊、需求不完整、需求与框架冲突 |
| 执行偏差 | Skill 定义 / 行为准则 | 行为偏差、约束违反、决策偏差 |

分批处理机制（避免上下文爆满）：

协调者按日志存在性调度子代理（最多 3 个并行）：
- ooda 分析子代理：分析 ooda-coder 日志（仅当存在时调度）
- 文档与调试分析子代理：分析 lb-doc-owner + debugger 日志（仅当存在时调度）
- 审查与架构分析子代理：分析 reviewer + architecture-guard 日志（仅当存在时调度）

分析索引机制：

执行日志目录下维护 `consolidation-index.md`，记录上次分析位置，避免重复分析。

**输出结构**：
- 问题频次分析（按 7 类分类统计）
- 系统性问题识别（跨 skill 的共性问题）
- 成熟度评估（半自治 / 磨合中 / 紧密控制）
- 演化建议（INDEX / CONVENTIONS / Skill 定义 / 行为准则 / 用户输入）
- 行业标准提升（从高频问题中提炼已经「超越当前标准」的实践）
- 决策建议（立即执行 / 需要决策 / 持续观察 / 成熟度建议）

**与开发者的关系**：开发者看到决策依据后，自行决定：
- 确认建议，调用 lb-doc-owner 执行文档变更？
- 再积累几个回合，观察问题是否持续？
- 这是 AI 的能力问题还是文档框架的问题？
- 哪些领域已经成熟，可以放手？哪些需要紧密控制？

## 工具协作关系

**核心原则**：
- 所有工具触发由开发者自行判断，节奏由开发者掌控
- 工具之间通过执行日志形成共演化闭环：执行 → 碎碎念（日志）→ 开发者决策 → 沉淀 → 更好的执行
- AI 发现问题只能汇报，不能自行修改文档
- consolidation 是闭环的关键节点，从执行失误中提取文档框架的缺陷，为开发者提供决策依据

```
开发者（唯一决策者）
        │
        ├── lb-doc-owner（统一文档维护）
        │     ├── 协调者：读取INDEX、检测项目状态、解析边界、维护项目级文档、生成执行日志
        │     ├── 初始化子代理（lb-bootstrap）：INDEX不存在时触发，搭建文档骨架
        │     ├── 业务层子代理：扫描业务模块，维护业务层文档
        │     └── 基础设施层子代理：扫描基础设施模块，维护基础设施层文档
        │
        ├── ooda-coder（执行 + 执行日志）
        │     ├── 协调者：调度子代理、注入行为准则、生成执行日志
        │     ├── 4个子代理：ooda-scout → ooda-forger → ooda-act → ooda-eval
        │     ├── 文件链路：context-brief → code-plan → execution-report
        │     ├── 渐进式披露：默认读直接输入，需要时沿链路回溯
        │     ├── 回溯机制：开发者可主动回溯到之前的阶段
        │     └── 输出执行日志目录（存放位置由INDEX配置）
        │
        ├── reviewer（独立审查）
        │     └── 读取文档 → 审查代码 → 输出报告 + 执行日志目录
        │
        ├── architecture-guard（架构守护）
        │     └── 读取 CONVENTIONS → 检查待合入代码 → 输出检查报告 + 执行日志目录
        │
        ├── debugger（问题诊断）
        │     └── 定位根因 → 输出诊断报告 + 调查过程记录 + 执行日志目录
        │
        └── consolidation（沉淀分析）
              ├── 读取所有 skill 的执行日志
              ├── 识别系统性问题模式
              ├── 评估领域成熟度
              ├── 生成决策依据（INDEX/CONVENTIONS/Skill定义/行为准则）
              └── 开发者决策 → lb-doc-owner 执行文档变更
```
