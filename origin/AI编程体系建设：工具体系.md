# AI编程体系建设：工具体系

本文档是 [AI编程方法论：骑士、缰绳和千里马](./AI编程方法论：骑士、缰绳和千里马.md) 的进一步细化，将理论框架落地为可操作的工具体系。

---

## 方法论核心（精炼提取）

### 三个角色

AI编程可以抽象为三个角色：

- **骑士（开发者）**：指挥官，决策者，最终责任人。拥有对项目的心智模型，是唯一有权修改缰绳的人。
- **缰绳（文档）**：骑士、AI、代码三者之间的介质，兼具约束（单向，规定边界）与信息交换（双向，达成共识）两种特性。
- **缰绳的不完备性**：缰绳是引导工具，帮助AI快速定位，避免大量阅读代码却找不到位置。有文档就用，没有文档就退化到纯代码探索模式。文档缺失时，AI应该继续工作，而不是纠结为什么没有文档。
- **千里马（AI）**：在约束范围内执行的行动者。拥有强大执行力但缺乏方向感，无法自我评估，上下文窗口有限。

### 核心机制

- **对齐**：通过文档让骑士和 AI 对项目达成共同理解，消除方向偏差。对齐是双向的，不是单方面指令。
- **行业标准**：约束文档不是凭空制定的，而是在实践中逐步演化。先有超越标准的实践，才会有标准的提升。
- **共演化**：骑士、缰绳、千里马三者相互磨合、共同进化。成熟领域可以半自治，新领域需要紧密控制。
- **回合制**：小回合（一次输入一次输出）是最小操作单元，若干小回合组成一个大回合（完整功能交付），项目在大回合中前进。
- **执行日志**：每个小回合结束后，AI 总结遇到的问题和困难，为骑士提供决策依据。
- **渐进式披露**：每个阶段读取最需要的文档，辅助类信息尽可能传递，避免上下文窗口过载。

### 核心原则

1. 方向优先于速度
2. 对齐优先于能力
3. 缰绳是活的，在共演化中持续进化
4. 质量可迁移——缰绳是模型无关的资产
5. 骑士是唯一有权修改缰绳的人
6. 生成与评估分离
7. 提示词会过时，缰绳不会

## 工具体系

基于上述方法论，工具体系分为五层：

```
基础层    —— 项目初始化
文档层    —— 建立缰绳（三个文档 owner）
执行层    —— 代码编写（内置执行日志）
校验层    —— 质量保障（独立审查 + 架构守护 + 问题诊断）
进化层    —— 共演化循环（沉淀分析）
```

---

## 基础层

### bootstrap

**职责**：项目初始化。在骑士接手一个新项目或老项目时，快速搭建文档框架。

**触发时机**：骑士接手项目时，一次性触发。

**两种场景**：

#### 场景一：新项目（从零开始）

**角色**：引导者。通过提问了解项目层级的问题，搭建最原始的项目层文档骨架。

**工作内容**：
- 提问了解项目：做什么、解决什么问题
- 确认技术选型：语言、框架、数据库等
- 确认架构风格：单体、微服务、前后端分离等
- 确认编码风格偏好（是否有参考规范）

**输出**：
- 项目介绍文档（骨架，不确定的留空）
- 编码规约（根据技术选型提供模板，骑士选择/修改）
- 项目级 INDEX（空，建议定义执行日志存放位置；如果骑士不愿意建立文件夹，后续 ooda-coder 可输出到控制台）
  - INDEX中应包含"AI执行记录"段落，配置路径为`.ai-work-logs/`

**注意**：不涉及业务模块划分，那是后续骑士在推进功能时逐步建立的。

#### 场景二：接手已有项目

**角色**：分析师。分析项目现状，输出分析报告，等骑士确认后再决定创建什么文档。

**工作内容**：
- 扫描项目结构，识别技术栈、目录组织
- 分析代码风格（命名、格式、注释习惯）
- 分析模块划分（哪些是基础设施、哪些是业务、哪些边界模糊）
- 识别公共工具函数

**输出**：
- 分析报告（不直接创建文档）

**后续流程**：
- 骑士审阅分析报告
- 骑士确认需要创建哪些文档
- 按确认内容创建，尽量少创建，避免文档爆炸
- 固定必要的：项目介绍、编码规约、项目级 INDEX
- 其他：视项目实际情况而定

**与三个 doc-owner 的关系**：bootstrap 负责"从无到有"，三个 doc-owner 负责"从有到优"。

## 文档层

### 三个 doc-owner 的关系

三个 doc-owner 不是从属关系，而是项目的各自一角，分别负责不同层级的文档：

```
┌─────────────────────────────────────┐
│              项目                    │
├─────────────┬─────────────┬─────────┤
│  项目层     │   业务层    │ 基础设施层 │
│             │             │          │
│ product-    │ business-   │ infra-   │
│ doc-owner   │ doc-owner   │ doc-owner│
└─────────────┴─────────────┴─────────┘
```

### 文档体系设计

**分层结构**：
- 项目层：项目根目录
- 业务层：业务相关目录（如 controller、services）
- 基础设施层：基础设施相关目录（如 dao、utils、model）

**文档类型**：
- README：介绍性质，"这是什么"
- INDEX：索引性质，"有什么、在哪"
- CONVENTIONS：规约性质，"怎么做"

**文档位置**：文档和代码放在一起，读代码时自然"发现"文档。

### 共同职责

- **建立**：根据各自管辖范围，创建 README、INDEX、CONVENTIONS
- **维护**：代码变更后、骑士指示时、发现问题时，更新文档
- **一致性检查**：函数签名是否同步、业务逻辑是否一致、命名规范是否符合
- **建议权**：分析代码结构，给出建议方案，必须得到骑士确认才能执行
- **汇报权**：发现文档有误时，不能自行修改，而是总结不同，让骑士决策

**工作原则**：没有任何 AI 比骑士更懂业务和架构。doc-owner 在初始化或补充文档时，先分析代码结构，给出建议方案，必须得到骑士确认才能执行。

### product-doc-owner

**职责**：项目级文档 owner。

**管辖范围**：

| 文档 | 位置 | 内容 |
|------|------|------|
| README | 项目根目录 | 项目是什么、解决什么问题、技术栈、依赖关系（如 controller → services → dao，禁止反向依赖） |
| INDEX | 项目根目录 | 模块目录：模块列表、位置、职责（如 controller - 业务入口，接收请求；services - 业务核心，处理逻辑）；AI执行记录：路径配置（如 `.ai-work-logs/`） |
| CONVENTIONS | 项目根目录 | 全局编码规约：命名规范（如类名 PascalCase、方法名 camelCase）、代码风格、禁止用法（如禁止在 controller 中直接调用 dao）；**硬性规定**：如果 AI 认为需要改动骑士未明确说明的相关地方，必须经过骑士同意 |

**特殊要求**：全局编码规约是整个项目的"行业标准"，修改必须由骑士明确授权。

### business-doc-owner

**职责**：业务级文档 owner。

**管辖范围**：

| 文档 | 位置 | 内容 |
|------|------|------|
| CONVENTIONS | 业务目录（如 services/） | 业务编码规约：业务特有的约束（如特定的设计模式、数据校验规则、业务流程规范） |

**特殊要求**：
- 业务层不需要 INDEX，业务文件按业务逻辑组织，直观易找（如 PermissionService、UserService 直接表明职责）
- 业务文档必须与代码中的实际业务逻辑一致
- 当代码变更导致业务逻辑变化时，需要同步更新文档

### infra-doc-owner

**职责**：基础设施文档 owner。

**管辖范围**：

| 文档 | 位置 | 内容 |
|------|------|------|
| 全局 INDEX | 基础设施根目录（如 utils/） | 工具目录：工具种类（如 JwtUtils - Token 工具、PermissionChecker - 权限校验工具）、简单介绍、指向模块 INDEX |
| 模块 INDEX | 各模块目录（如 utils/） | 函数签名：工具名称、函数名、入参、出参（如 `check(Long userId, String resource, String action) → boolean`） |

**特殊要求**：
- 函数签名 INDEX 是基础设施层的核心资产，必须与代码中的实际函数签名保持同步
- INDEX 设计为树状结构：全局 INDEX → 模块 INDEX
- 全局 INDEX 告诉 AI 有哪些工具模块，模块 INDEX 告诉 AI 函数怎么调用

## 执行层

### ooda-coder

**职责**：基于 OODA-E 循环进行代码编写。

**核心价值观**：规范大于自由，基建优先于手写，质量兜底一切。

**架构概览**：

ooda-coder 通过 4 个 subagent 实现上下文隔离，确保弱模型也能保持最优能力。

| 角色 | OODA-E阶段 | 职责 |
|------|-----------|------|
| ooda-scout | Observe+Orient | 读取文档，输出上下文快照 |
| ooda-forger | Decide+Draft+Verify | 生成代码+静态预检 |
| ooda-act | Act | 写入文件+编译+基础功能验证 |
| ooda-eval | Evaluate | 回归测试+覆盖率分析+补充测试 |
| 主session | 执行日志 | 汇总过程+识别卡点+询问用户 |

**临时文件机制**：

主session负责临时文件的完整生命周期。每次执行创建唯一sessionId（UTC时间戳），临时文件存放在`.ooda-work/{sessionId}/`。

主session创建目录、分配写入路径、管理清理。subagent只负责写入，不管理目录。

| 环节 | 写入 | 读取 |
|------|------|------|
| 主session | 创建/删除目录 | - |
| ooda-scout | context-brief.md | - |
| ooda-forger | code-plan.md | context-brief.md |
| ooda-act | - | code-plan.md |
| ooda-eval | - | - |

降级策略：subagent写入失败时，在返回中输出完整内容。主session根据返回值判断传递方式。

**OODA-E 流程**：

1. **ooda-scout（侦察）**：业务上下文与基建环境侦察
   - 必读项目索引文件（INDEX或等效文件），获取项目结构和模块位置
   - 如果项目索引文件不存在，按降级链读取：README → 根目录md文件 → 扫描代码目录结构
   - 根据任务复杂度和意图清晰度判断是否需要额外读取README/架构文档
   - 读取业务文档，获取业务上下文
   - 编码规约只记录路径，不读取内容
   - 一切以代码为准，文档与代码有差异时只提示不修改
   - 文档缺失时继续探索代码，只提示不阻塞

2. **ooda-forger（规划）**：规范与依赖对齐，制定执行计划
   - 从CONTEXT-BRIEF获取项目结构、规约路径、业务背景、架构约束
   - 必读编码规约内容
   - 从代码中提取依赖的API签名
   - 拟稿阶段，不能完全相信文档描述，依赖的功能必须查询代码确认
   - 如果对业务无法理解，必须停下来询问开发者

3. **ooda-act（执行）**：代码编写与验证
   - 从CODE-PLAN获取文件路径、完整代码、验证命令
   - 从主session获取规约路径和项目结构（仅自修复模式使用）
   - 默认模式（机械执行）：不读取编码规约，只信任forger的CODE-PLAN
   - 自修复模式（编译失败时激活）：可读取编码规约、相关源码、INDEX等，探索范围仅限于修复当前编译错误所需
   - 写入代码并执行编译检查
   - 如果编译失败，自行修正并重新验证，最多3次
   - 如果3次修正后仍失败，停止并报告

4. **ooda-eval（评估）**：回归测试与覆盖率分析
   - 从CONTEXT-BRIEF获取项目结构、规约路径
   - 必读编码规约内容、测试代码
   - 使用git diff获取实际变更的文件列表，结合用户意图和act的修改描述进行交叉验证
   - 测试失败或交叉验证发现差异时，先分析原因：
     - 功能bug：自行修复
     - 测试样例过时：自行更新测试样例
     - 业务不清楚：询问开发者
     - 改动点明显影响其他功能：询问开发者
   - 小bug、逻辑问题自行解决，问题较大或不清楚时询问开发者

**回溯机制**：

回溯是开发者的明确指令。当流程出问题停止时，主session可以建议开发者使用回溯指令，但最终决定权在开发者。

- 回溯指令格式：`回溯到 [节点名称]`
- 可选节点：scout、forger、act、eval
- 回溯时保留之前阶段的输出（如CONTEXT-BRIEF）

**执行日志**：

全部subagent执行完毕后，主session负责汇总各subagent的交互过程，生成执行日志。

读取项目级INDEX，查找"AI执行记录"段落中的"路径"配置：
- 有配置：自动写入，不询问
- 无配置：询问开发者是否保存
  - 是：让开发者输入路径，写入，提示在INDEX上配置以后都会生效
  - 否：不记录

执行日志格式：
```markdown
# 执行日志

## 任务信息
- 时间戳：{UTC秒级时间戳}
- 时间：{YYYY-MM-DD HH:mm:ss UTC}
- 用户输入：{完整的用户输入，如果使用skill只记录用户输入内容}
- 使用skill：{skill名称，无则写"无"}

## 执行过程
- ooda-scout：{顺利/卡点，具体描述}
- ooda-forger：{顺利/卡点，具体描述}
- ooda-act：{顺利/卡点，具体描述}
- ooda-eval：{顺利/卡点，具体描述}

## 卡点纪要
[只记录问题，顺利的不记录。没有则写"无"]
```

日志文件命名：`{UTC时间戳}-任务描述.md`（任务描述不超过10个字）

日志文件夹不存在时，如果开发者明确要求或INDEX有明确指定，自动创建。

**硬性规定**：
- 如果 AI 认为需要改动骑士未明确说明的相关地方，必须经过骑士同意
- 禁止手写基础设施已有的能力，必须调用 INDEX 中提供的 API
- 所有subagent无权修改文档，发现不一致时汇报给主session

**与缰绳的关系**：
- ooda-coder 在缰绳的约束下执行，无权修改缰绳
- AI 发现文档有误时，不能自行修改，而是总结不同，让骑士决策

**文档读取策略**：

渐进式披露，每个阶段读取最需要的文档，辅助类信息尽可能传递：

**设计原则**：
- INDEX（方向标）贯穿始终，每个阶段都需要知道项目结构
- 编码规约在侦查阶段只记录路径，规划阶段读取内容，编码阶段默认不读取（编译失败时按需读取）
- 辅助类信息尽可能通过CONTEXT-BRIEF传递，避免重复读取

| 文档类型 | ooda-scout | ooda-forger | ooda-act | ooda-eval |
|----------|------------|-------------|----------|-----------|
| INDEX（方向标） | 读取 | 从快照获取 | 从快照获取 | 从快照获取 |
| 编码规约 | 只记录路径 | 读取内容 | 默认不读取，编译失败时按需读取 | 读取内容 |
| README/架构文档 | 按需读取 | 不需要 | 不需要 | 不需要 |
| 目标文件 | 不需要 | 不需要 | 读取 | 读取 |
| 测试代码 | 不需要 | 不需要 | 不需要 | 读取 |

文件树优化：
- 有全局INDEX时：基础设施层指向全局INDEX，业务层指引到涉及的模块目录
- 无全局INDEX时：指引规划agent到正确的目录位置

## 校验层

### reviewer

**职责**：独立审查 ooda-coder 的产出，提供客观的质量评估。

**核心原则**：
- 生成与评估必须分离
- 所有审查维度必须客观可验证
- 只输出报告，不做修改决定
**触发时机**：由开发者自行判断。

**工作流程**：
1. **独立 session 启动**：干净的上下文，不受 ooda-coder 的上下文影响
2. **读取项目文档**：README、INDEX、CONVENTIONS（全局 + 业务）
3. **读取代码变更**：获取 ooda-coder 的代码产出
4. **按维度审查**：逐项检查 14 个审查维度
5. **输出审查报告**：结构化报告，包含各维度结果和总体结论

**审查维度（14 个）**：

#### 基础合规维度
1. **编码规约符合度**：对照 CONVENTIONS，检查命名规范、代码风格、禁止用法等
2. **架构合规性**：检查依赖方向、模块边界是否符合架构约束
3. **基础设施使用**：检查是否正确调用 INDEX 中的 API，是否复用了已有能力
4. **业务逻辑一致性**：对照业务文档，检查代码中的业务逻辑是否一致

#### 质量维度
5. **代码质量**：可读性、可维护性、边界处理、异常处理等客观标准
6. **变更影响分析**：修改了哪些函数/类、这些函数/类在项目中的其他使用点、变更是否可能影响其他功能、是否需要更新相关测试
7. **可扩展性**：是否易于扩展，是否遵循开闭原则
8. **可测试性**：是否易于测试，是否依赖注入、可 mock
9. **性能考虑**：潜在性能问题，如 N+1 查询、不必要的循环
10. **安全性**：安全漏洞，如 SQL 注入、XSS
11. **错误处理**：异常处理是否恰当，是否有未捕获的异常
12. **技术债务**：是否引入技术债务，是否有临时解决方案
13. **设计模式使用**：是否使用了合适的设计模式，是否过度设计
14. **接口设计**：API 设计是否合理，是否易于理解和使用

**输出格式**：

```markdown
# reviewer 审查报告

## 审查摘要
- 审查范围：[文件列表]
- 审查时间：[时间]
- 总体结论：[通过/有条件通过/不通过]

## 各维度审查结果
[每个维度包含：状态、问题描述、严重程度]

## 变更影响分析
- 修改的函数/类：[列表]
- 其他使用点：[列表]
- 回归风险：[高/中/低]
- 建议的测试：[列表]

## 改进建议
- [建议列表]

## 总体结论
[通过/有条件通过/不通过] + 理由
```

**与骑士的关系**：reviewer 只输出报告，不做修改决定。修改决定由骑士做出。

### architecture-guard

**职责**：架构约束的机械化检查，确保代码合入前符合架构规范。

**核心原则**：
- 有些约束不能只靠文档"建议"，必须被确定性地检查
- 检查规则来源于项目级文档和部门级文档
- 只输出报告，不做修改决定

**触发时机**：由开发者自行判断。

**工作流程**：
1. **读取架构约束**：从项目级文档和部门级文档中读取
2. **解析代码变更**：获取待合入的代码 diff
3. **执行检查规则**：按维度逐项检查
4. **输出检查报告**：通过/不通过 + 具体违规项

**检查维度（13 个）**：

#### 架构约束维度
1. **依赖方向**：检查 import 语句，确保没有反向依赖
2. **模块边界**：检查是否直接访问其他模块的内部类/方法
3. **分层架构**：检查调用链是否符合分层架构
4. **禁止用法**：检查是否使用了禁止的 API 或模式，安全检查（具体检查内容参考项目级编码规约）

#### 变更管理维度
5. **接口兼容性**：检查是否修改了现有的 API 接口（方法签名、返回类型等）
6. **数据库变更**：检查是否有数据库 schema 变更（如 DDL 语句）
7. **配置变更**：检查是否有配置文件变更
8. **依赖变更**：检查是否有新的依赖（如 pom.xml、package.json 变更）

#### 质量保障维度
9. **测试覆盖**：检查是否有对应的测试代码
10. **文档更新**：检查是否需要更新文档（如 API 文档、业务文档）

#### 文档规范维度
11. **重复文档检查**：检查是否有重复的文档内容
12. **文档位置检查**：检查文档是否放在正确的位置
13. **文档必要性检查**：检查是否创建了不必要的文档

**检查规则来源**：
- **依赖方向、模块边界、分层架构**：从 README 中的架构描述提取
- **禁止用法**：从 CONVENTIONS 中的禁止用法章节提取
- **安全检查**：从 CONVENTIONS 中的安全规约提取
- **接口兼容性**：从 INDEX 中的函数签名提取
- **数据库变更、配置变更、依赖变更**：从代码 diff 中识别
- **测试覆盖**：检查测试目录中是否有对应的测试文件
- **文档规范**：从 doc-owner 的文档规范提取

**输出格式**：

```markdown
# architecture-guard 检查报告

## 检查摘要
- 检查范围：[文件列表]
- 检查时间：[时间]
- 总体结论：[通过/不通过]

## 各维度检查结果
[每个维度包含：状态、违规项描述]

## 违规项详情
- [违规项]：[文件:行号] - [违规描述]

## 总体结论
[通过/不通过] + 理由
```

**与 reviewer 的区别**：
- reviewer 关注"代码写得好不好"——是质量审查
- architecture-guard 关注"代码有没有违反架构约束"——是合规检查

**与 product-doc-owner 的关系**：architecture-guard 的检查规则来源于 product-doc-owner 管辖的项目级文档。如果架构约束需要调整，由骑士授权 product-doc-owner 修改。

### debugger

**职责**：系统化调试。当出现 bug、测试失败、构建错误等代码问题时，通过严谨的调查流程定位根因，输出诊断报告。

**核心原则**：
- 先定位根因，再提出修复方案
- 诊断与修复分离——只输出诊断结论，不做修改决定
- 调试是发现系统问题的最佳时机，执行日志为沉淀分析提供数据来源

**触发时机**：由开发者自行判断。出现代码缺陷、测试失败、构建错误、异常行为时触发。

**工作流程**：
1. **快速定位**：有文档看文档快速导航，没文档扫描代码目录
2. **根因调查**：仔细阅读错误信息、稳定复现、检查近期变更、追踪数据流
3. **模式分析**：找到可用示例、对比差异、理解依赖
4. **假设验证**：形成单一假设、最小化验证、科学方法
5. **诊断结论**：输出结构化诊断报告
6. **执行日志**：记录调试纪要，写入执行日志目录

**诊断报告**：
```markdown
# debugger 诊断报告

## 问题摘要
- 问题描述/复现步骤/复现率/影响范围

## 根因分析
- 故障点/根因/证据链/触发条件

## 模式分析
- 可用示例/关键差异/依赖关系

## 修复建议
- 建议方案/改动范围/风险评估/验证方式

## 待确认事项/总体结论
```

**执行日志**：

调试完成后生成执行日志，记录调试纪要（问题、根因、根因类型、调查耗时），为 consolidation 提供数据来源。日志路径读取项目级 INDEX 中"AI执行记录"段落的配置。

**与 reviewer 的区别**：
- reviewer 审查的是"代码写得好不好"——在代码已经写完之后做质量审查
- debugger 诊断的是"代码为什么出了问题"——在问题发生后定位根因

**与 ooda-coder 的关系**：
- ooda-coder 负责写代码，debugger 负责诊断问题
- debugger 的诊断报告是 ooda-coder 修复问题的决策依据
- 两者都生成执行日志，共同为 consolidation 提供数据

## 进化层

### consolidation

**职责**：提取、分类、统计和总结执行日志的积累，输出结构化分析报告。

**核心原则**：
- consolidation 是分析器，不是执行器
- 只负责分析和呈现，不直接修改文档
- 输出结果，由其他角色或人工决策后续动作
- 根据日志内容自行判断流程和卡点，不绑定特定流程

**触发时机**：由开发者自行判断。

**工作流程**：
1. **读取执行日志目录**：从项目级 INDEX 中"AI执行记录"段落获取执行日志存放位置
2. **解析执行日志内容**：读取所有执行日志文件，提取结构化信息：
   - 任务信息：时间戳、时间、用户输入、使用skill
   - 执行过程：根据日志内容自行判断流程和卡点
   - 卡点纪要：记录的问题
3. **按分类统计频次**：统计每个问题分类的出现次数
4. **识别重复模式**：识别重复出现的问题模式
5. **分析根本原因**：深入分析问题背后的根本原因
6. **输出结构化报告**：生成分析报告

**分析维度**：

#### 频次统计
按问题分类统计出现次数。

#### 模式识别
1. **高频问题**：反复出现的问题，说明缰绳在这些地方不够清晰
2. **问题聚集**：某些模块或功能的问题集中出现
3. **时间趋势**：问题是否随时间减少，说明缰绳在改进

#### 根因分析
1. **文档缺失**：缺少必要的文档指引
2. **文档模糊**：文档描述不够清晰明确
3. **约束不足**：缺少必要的约束规则
4. **能力边界**：AI 在某些领域的固有局限
5. **上下文不足**：提供的上下文信息不够

**输出格式**：

```markdown
# consolidation 分析报告

## 分析摘要
- 分析范围：[执行日志文件列表]
- 执行日志总数：[数量]
- 问题总数：[数量]
- 时间跨度：[最早日志时间 - 最近日志时间]

## 高频问题清单
[每个问题包含：问题分类、出现次数、占比、典型表现、影响程度]

## 问题模式识别
[每个模式包含：模式名称、描述、涉及问题、出现频率]

## 根因分析
[每个根因包含：根因名称、描述、影响范围、严重程度]

## Skill使用分析
- 使用skill的日志数量：[数量]
- 未使用skill的日志数量：[数量]
- 高频使用的skill：[列表]

## 能力边界画像
- AI 表现良好的领域：[列表]
- AI 频繁出错的领域：[列表]

## 约束缺失提示
[可以通过补充约束来避免的问题列表]

## 文档盲区提示
[文档指引不够导致 AI 反复搜索的模块列表]

## 改进建议
- 短期改进（立即可做）：[列表]
- 中期改进（需要积累）：[列表]
- 长期改进（需要验证）：[列表]
```

**与骑士的关系**：骑士看到分析报告后，自行决定：
- 现在就通过对应的 doc-owner 更新文档？
- 再积累几个回合，观察问题是否持续？
- 这是 AI 的能力问题还是缰绳的问题？

## 工具协作关系

**核心原则**：
- 所有工具触发由开发者自行判断，节奏由开发者掌控
- 工具之间没有直接协作，所有结果都回流到开发者
- AI 发现问题只能汇报，不能自行修改缰绳

```
开发者（唯一决策者）
        │
        ├── bootstrap（初始化）
        │     └── 生成文档骨架（包含INDEX中的AI执行记录配置）
        │
        ├── product-doc-owner ──┐
        ├── business-doc-owner ──┤── 建立和维护缰绳（各司其职，无从属关系）
        ├── infra-doc-owner ────┘
        │
        ├── ooda-coder（执行 + 执行日志）
        │     ├── 4个subagent：ooda-scout → ooda-forger → ooda-act → ooda-eval
        │     ├── 渐进式披露：每个阶段读取最需要的文档
        │     ├── 回溯机制：开发者可主动回溯到之前的阶段
        │     └── 输出执行日志（存放位置由INDEX配置）
        │
        ├── reviewer（独立审查）
        │     └── 读取缰绳 → 审查代码 → 输出报告
        │
        ├── architecture-guard（架构守护）
        │     └── 读取 CONVENTIONS（项目级+部门级） → 检查待合入代码 → 输出检查报告
        │
        ├── debugger（问题诊断）
        │     └── 定位根因 → 输出诊断报告 + 执行日志
        │
        └── consolidation（沉淀分析）
              ├── 读取执行日志积累（包含 ooda-coder 和 debugger 的日志）
              └── 输出分析报告 → 开发者决策 → doc-owner 执行
```
