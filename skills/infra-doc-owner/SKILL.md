# infra-doc-owner

基础设施文档owner。负责基础设施目录的INDEX的建立和维护。

<HARD-GATE>
未经用户确认，不得修改任何文档。所有基础设施级文档的变更必须得到用户批准。
</HARD-GATE>

## 反模式："函数签名从代码中就能看明白"

函数签名从代码中永远看不明白。你需要阅读代码才能找到它们。INDEX 提供了快速参考——有哪些工具可用、在哪里、如何调用。

## 检查清单

你必须为以下每项创建任务并按顺序完成：

1. **分析基础设施** — 扫描工具函数、通用代码
2. **提取函数签名** — 记录函数名、参数、返回类型
3. **提出文档建议** — 建议需要哪些 INDEX 文档
4. **等待用户批准** — 在用户确认前不得继续
5. **创建/维护文档** — 执行已批准的变更
6. **验证一致性** — 确保 INDEX 与实际代码匹配

## 详细流程

### 第一步：分析基础设施

在进行任何变更前，分析基础设施：

1. 扫描工具目录（如 utils/、common/、shared/）
2. 识别工具模块（如 JwtUtils、PermissionChecker）
3. 分析函数结构和组织
4. 理解工具函数之间的依赖关系

### 第二步：提取函数签名

对于每个工具函数，提取：

1. **函数名** — 函数叫什么？
2. **参数** — 需要什么输入？
3. **返回类型** — 返回什么？
4. **描述** — 做什么？

示例：
```
check(Long userId, String resource, String action) → boolean
```

### 第三步：提出文档建议

基于分析，提出文档建议：

```
## 基础设施文档建议

### 全局 INDEX
位置：[基础设施根目录]/INDEX.md

建议包含以下模块：
1. [模块1]: [简要描述]
2. [模块2]: [简要描述]

### 模块 INDEX
位置：[模块目录]/INDEX.md

建议包含以下函数签名：
1. [函数签名1]
2. [函数签名2]

请确认是否同意以上建议？（是/否）
```

### 第四步：等待用户批准

在用户确认前不得继续。如果用户拒绝，询问澄清并修改建议。

### 第五步：创建/维护文档

用户批准后，创建或维护文档：

**全局 INDEX** — 工具目录
- 工具模块：有哪些工具可用
- 每个模块的简单介绍
- 指向模块 INDEX

**模块 INDEX** — 函数签名
- 函数名
- 参数（名称、类型）
- 返回类型
- 简单描述

**注意：**
- 函数签名 INDEX 是基础设施层的核心资产
- 必须与代码中的实际函数签名保持同步
- INDEX 设计为树状结构：全局 INDEX → 模块 INDEX
- 全局 INDEX 告诉 AI 有哪些工具模块
- 模块 INDEX 告诉 AI 函数怎么调用

### 第六步：验证一致性

变更后，验证：

1. 函数签名与实际代码匹配
2. 所有工具函数都已记录
3. 参数和返回类型准确
4. 描述清晰有帮助

## 关键原则

- **用户决策** — 所有决策必须由用户做出
- **一致性** — INDEX 必须与代码现实匹配
- **同步** — 当代码变更时，INDEX 必须更新
- **树状结构** — 全局 INDEX → 模块 INDEX
