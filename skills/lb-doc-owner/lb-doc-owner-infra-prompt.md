## INDEX 结构说明

基础设施层 INDEX 为树状结构：
- **全局 INDEX**：位于基础设施根目录，列出各工具模块的名称、路径、一句话说明
- **模块 INDEX**：位于各模块目录，列出函数签名（函数名、参数、返回类型）

## 分析指引

### 1. 探索模块文档

进入每个管辖模块的目录，查找是否存在 README、INDEX、CONVENTIONS 文档。不要假设文档不存在，必须实际探索确认。同时检查是否存在全局 INDEX 及其指向的模块 INDEX。

### 2. 分析基础设施

- 扫描工具函数、通用代码、共享模块
- 识别模块的职责分类（如 JWT 工具、权限校验、数据转换、日志工具等）
- 理解模块间的依赖关系

### 3. 提取函数签名

对每个工具模块，提取以下信息：

- **函数名**：完整的方法名
- **参数列表**：参数名、参数类型、是否可选、默认值
- **返回类型**：返回值的类型
- **异常信息**：可能抛出的异常（如有）

示例：
```
check(Long userId, String resource, String action) → boolean
generateToken(Map<String, Object> claims, long expiration) → String
parseToken(String token) → Claims (throws JwtException)
```

### 4. 维护 INDEX 导航链

检查 INDEX 的导航完整性：

- **全局 INDEX**：是否指向所有模块 INDEX？条目是否包含模块名、路径、一句话说明？
- **模块 INDEX**：函数签名是否与代码一致？是否有遗漏或过时的签名？
- **信息承载**：有 README 的模块，INDEX 只需定位；无 README 的模块，INDEX 的一句话说明是否足够让 AI 判断是否需要深入探索？

### 5. 验证文档一致性

对已存在的文档逐项检查：

- **全局 INDEX**：模块列表是否完整？路径是否正确？描述是否准确？
- **模块 INDEX**：函数签名是否与代码中的实际签名一致？参数类型、返回类型是否准确？
- **README**（如存在）：描述是否与模块实际职责一致？

### 6. 证据采集

声称文档与代码一致或不一致时，必须提供证据：

- INDEX 中的函数签名 → 对应代码中的函数定义（文件名:行号）
- 模块描述 → 对应代码中的模块入口文件
- 导航路径 → 实际可访问性验证
